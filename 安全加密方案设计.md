# 安全加密方案设计

## 目标

本方案覆盖两类安全需求：

1. **通信链路加密**：降低传输过程被抓包后泄露业务数据与密钥的风险。
2. **数据库敏感字段加密**：即使数据库文件被拷走，也不能直接看到原始文案、翻译结果与错误详情。

## 一、通信链路加密方案

### 1.1 传输层总体策略

- 前端部署在 Vercel，默认使用 HTTPS。
- 后端部署在 Render，默认使用 HTTPS。
- 前端只通过 `https://` 的后端地址访问 API。
- 后端只通过 HTTPS 调用 OpenAI-compatible 接口与 ElevenLabs 接口。

### 1.2 后端安全控制

在 FastAPI 中增加以下能力：

- `HTTPSRedirectMiddleware`：当 `BACKEND_URL` 为 HTTPS 时，强制重定向到 HTTPS。
- 安全响应头：
  - `Strict-Transport-Security`
  - `X-Frame-Options: DENY`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
- 严格 CORS：只允许受信前端域名访问。

### 1.3 部署要求

- Vercel 中 `VITE_API_BASE_URL` 必须填写 Render 的 HTTPS 地址。
- Render 中 `FRONTEND_URL` 必须填写 Vercel 的 HTTPS 地址。
- 浏览器访问应统一使用正式域名，不再使用 HTTP 直连。

## 二、数据库敏感字段加密方案

### 2.1 加密范围

建议加密以下字段：

- `generation_jobs.source_text`
- `generation_jobs.translated_text`
- `generation_jobs.error_message`
- `voice_outputs.error_message`

说明：

- `voice_id`、`voice_name`、`status`、时间戳不属于高敏信息，可保持明文，便于调试与展示。
- 音频文件本身仍保存在本地目录，当前版本不做文件级加密。

### 2.2 加密方式

采用 **应用层对称加密**：

- 算法库：`cryptography`
- 方案：`Fernet`
- 密钥来源：环境变量 `FIELD_ENCRYPTION_KEY`

优点：

- 实现简单，适合现有 FastAPI + SQLite 架构
- 自带认证能力，能检测密文被篡改
- 不需要修改数据库字段类型

### 2.3 读写流程

写入时：

1. 业务代码拿到原文
2. 调用 `encrypt_text()`
3. 将密文写入数据库

读取时：

1. 从数据库取出密文
2. 调用 `decrypt_text()`
3. 返回给前端明文

### 2.4 兼容旧数据

为避免影响当前已存在的明文历史数据：

- 新增统一前缀：`enc::`
- 解密时先判断是否带此前缀
- 没有前缀则视为旧明文，直接返回

这样可以做到：

- 新数据自动加密
- 旧数据不迁移时也能继续读取
- 迁移脚本可在后续独立执行

### 2.5 历史搜索策略

字段加密后，数据库无法直接对密文做 `LIKE` 模糊查询。

当前项目采用：

- 先取出任务列表
- 对 `source_text`、`translated_text` 解密
- 在 Python 内存中执行关键字筛选

适用场景：

- 当前笔试/演示规模的数据量较小

后续若数据量增大，可升级为：

- 额外维护脱敏搜索索引字段
- 接入 Elasticsearch / OpenSearch

## 三、错误信息保护

第三方接口报错时，不应将完整原始响应直接落库或返回前端。

改造原则：

- 只保留必要摘要：`status_code`
- 尽量附带请求追踪号：`request_id`
- 不回传第三方原始报文全文

示例：

- 推荐：`翻译模型请求失败，status=429，request_id=abc123`
- 不推荐：直接保存第三方整段错误 JSON

## 四、密钥管理要求

### 4.1 环境变量

新增：

```env
FIELD_ENCRYPTION_KEY=
```

### 4.2 生成方式

使用 Fernet 标准密钥：

```bash
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

### 4.3 管理建议

- 本地开发：写入 `backend/.env`
- Render：配置为服务环境变量
- 不提交到 GitHub
- 如密钥曾泄露，应重新生成并重新加密数据

## 五、迁移方案

为兼容已有明文历史数据，提供单独迁移脚本：

```bash
cd backend
python -m scripts.migrate_encrypt_existing_jobs
```

迁移脚本会：

- 扫描历史任务
- 识别未加密字段
- 自动转成密文
- 保留已加密数据不重复处理

## 六、当前方案边界

当前版本已覆盖：

- Web 传输加密
- API 到第三方服务的 HTTPS 调用
- 数据库敏感文本字段加密
- 错误摘要脱敏

尚未覆盖：

- 音频文件本身加密
- 密钥轮换自动化
- 更细粒度的用户鉴权
- 数据库搜索索引加密优化

## 七、结论

这套方案适合当前项目阶段，优点是：

- 改造成本低
- 对现有数据库结构侵入小
- 能满足笔试项目对“通信安全 + 落库保护”的核心要求
- 保留后续扩展空间
